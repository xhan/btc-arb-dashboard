# 套利路径设计（当前实现现状，便于检索与升级）

## 文档目的

记录当前“套利路径”功能的实现思路、配置方式与关键约束，方便后续：

- 快速检索当前行为（为什么会这样显示）
- 增加新固定路径规则（如新增稳定币/封装币对）
- 升级搜索策略（路径深度、去重、规则边策略、费用模型）
- 排查展示结果与用户认知不一致的问题

- 固定路径：按预定义规则计算（每条规则输出 1 条最佳机会）

## 设计目标（现状）

当前实现优先解决的是“可用、可读、可维护”，而不是精确成交模拟：

- 能从实时报价中快速构图并搜索闭环
- 允许规则币（等价值币）参与闭环，提升可发现机会数量
- 避免明显重复机会（同一路径不同起点）
- 展示层尽量简洁（隐藏规则腿、优先从 `cbBTC` 一类起点显示）
- 固定路径配置可扩展（写死链对 / 跨链自动选最优两种模式）

## 核心数据模型

### 1) 边（Edge）

搜索统一在“符号图”上进行，边结构形如：

```js
{ from, to, rate, chain, quoteId?, inverse?, rule? }
```

来源分两类：

- 报价边：由监控报价生成（真实交易方向）
- 规则边：由别名规则生成（1:1 等价映射，如 `BTC.b <-> cbBTC`）

当前 symbol 语义：

- 路径图按原始 symbol 字符串精确匹配
- `wBTC` 与 `WBTC` 视为不同 symbol
- 只有显式等价资产规则才允许不同 symbol 之间生成规则边
- 不再在搜索核心里做全局大小写归一

### 2) 路径 / 闭环（Cycle）

搜索结果统一表示为：

```js
{ legs: Edge[], profitRate }
```

- `legs`：闭环路径上的每一条边
- `profitRate`：所有 `rate` 连乘后减 1（未扣 gas / 滑点 / 跨链成本）

## 数据流（从报价到面板）

### 1) 报价数据 -> 图边

`app.js` 在 `updateArbPanel()` 中读取：

- `dashboardState`（分区与报价列表）
- `quoteMonitorState`（报价状态，含 `fromSymbol/toSymbol/lastRawPrice` 等）

再通过 `window.ArbPaths.buildEdges(...)` 生成报价边。

说明：

- 若报价支持反向显示（`showInverse`），会补一条反向边（使用 `inverseRawPrice`）
- 搜索深度与展示逻辑都基于这些边进行

### 2) 别名规则 -> 规则边（1:1 等价币）

当前不再直接手写零散 `aliasRules`，而是先在 `app.js` 里维护 `ASSET_EQUIVALENCE_GROUPS`，再生成别名规则。

当前示例：

- `cbBTC` 组：`cbBTC / xBTC / BTCB / BTC.b / BTC.B`
- `tBTC` 组：`tBTC / TBTC`

注意：

- 这一层只处理 symbol 等价关系
- 不把 token address / Sui type address 直接塞进规则边
- 地址仍然作为报价元数据保留，用于详情展示和人工核对

这样做的原因是：当前套利图构建只依赖 `fromSymbol / toSymbol`，不依赖 token address。

`arb-paths.js` 的 `buildRuleEdges()` 会把同一等价组做成“组内全互联”：

- 不再只是 `alias <-> target` 星型
- 会直接生成 `BTC.b <-> xBTC` 这种规则边

这样可以避免路径里出现没有意义的连续两段规则跳转（如 `BTC.b -> cbBTC -> xBTC`）。

## 搜索算法（当前实现）

### 1) 搜索范围

分区路径与全局路径都使用 `findTopCycles()` 搜索闭环。

参数（当前前端）：

- `maxDepth: 4`（最多 4 腿）
- 分区 `limit: 4`
- 全局 `limit: 8`
- `acceptCycle: isMeaningfulPath`（过滤纯规则路径）

### 2) 搜索方式

`findTopCycles()` 使用 DFS：

- 建立邻接表 `from -> outgoing edges[]`
- 从每个起点出发尝试回到起点形成闭环
- 使用 `visited` 避免重复访问 symbol（闭环收尾回到起点除外）

### 3) 规则边约束（重要）

当前规则边策略是：

- 规则边参与搜索与收益计算（rate=1）
- 规则边**仍计入**路径步数（`maxDepth`）
- 禁止连续规则边（含闭环首尾相邻）
- 纯规则路径不展示（`isMeaningfulPath`）

这能兼顾可发现性与可读性，避免规则腿主导路径结构。

### 4) 去重与排序

同一闭环从不同起点出发会被 DFS 枚举多次。当前通过“旋转归一化”去重：

- 对 `legs` 做所有旋转版本
- 选一个 canonical key 作为闭环唯一标识
- 若已出现则不重复收录

排序规则：

- 按 `profitRate` 从高到低排序

## 展示层策略（当前实现）

### 1) 规则腿不显示（但参与计算）

`renderOpportunity()` 渲染前会过滤 `rule` / `chain === '规则'` 的腿：

- 解决 `(规则)` 行干扰阅读的问题
- 收益率仍按完整路径（含规则边）计算

### 2) 起点偏好（优先从 cbBTC/等价币开始显示）

为了避免同一路径从 `WBTC` 或其他别名开始显示，前端给 `findTopCycles()` 传入 `preferredStartSymbols`：

- `cbBTC`
- `xBTC`
- `BTCB`
- `BTC.b`
- `BTC.B`

闭环去重时会优先选择这些 symbol 作为展示起点（如果存在）。

### 3) 展示数量规则（分区 / 全局）

当前显示数量不是简单固定值，而是“正收益优先”的策略：

- 分区路径：
  - 最多显示 `4` 条
  - 仅展示 `profitRate > 0` 的路径
  - 如果没有正收益，但有路径，则至少显示 `1` 条（收益最高）
- 全局路径：
  - 最多显示 `8` 条
  - 仅展示 `profitRate > 0` 的路径
  - 如果没有正收益，但有路径，则至少显示 `1` 条（收益最高）

### 4) 全局路径过滤（当前实现）

全局路径区域提供独立过滤栏，当前有两种过滤：

- 代币过滤：按路径腿里的 `from / to` symbol 精确匹配
- 链过滤：按路径腿里的 `chain` 匹配

当前 UI 约束：

- 代币过滤和链过滤在同一行
- 仅作用于“全局路径”列，不影响固定路径和分区路径
- 输入值使用空格分隔多个 token，不再支持逗号分隔

链过滤的匹配规则是“精确别名映射”，不是模糊匹配，也不是前缀匹配：

- 先尝试精确匹配内部链 key（如 `ethereum`）
- 再尝试精确匹配显示名（如 `Arbitrum`、`ETH`）
- 再尝试匹配预定义简称别名（如 `ARB` -> `arbitrum`、`OP` -> `optimism`、`AVAX` -> `avalanche`）

命中后会统一归一化为内部链 key，再用于过滤路径。

这样做的目的：

- 避免模糊匹配带来的误伤
- 保持输入体验（支持常见简称）
- 让后续维护只需要扩展别名表，而不需要改过滤算法

## 固定路径规则设计（可扩展）

固定路径走 `findBestFixedPath()`，当前支持两种模式（`steps: 2`）：

### 模式 A：写死链对（固定链组合）

适合明确链间对敲场景，例如：

```js
{
  title: 'WBTC ETH <-> ARB',
  base: 'cbBTC',
  quote: 'WBTC',
  chains: ['ethereum', 'arbitrum'],
  steps: 2
}
```

行为：

- 会比较 `ETH(base->quote)+ARB(quote->base)` 与反向链顺序
- 返回收益更高的一组

### 模式 B：固定币种对，跨链自动选最优（推荐扩展模式）

适合你这次新增的场景，例如：

```js
{
  title: 'GHO <-> USDC',
  base: 'GHO',
  quote: 'USDC',
  steps: 2,
  crossChain: true
}
```

行为：

- 在所有非规则链里枚举不同链组合 `chainA != chainB`
- 查找 `chainA: base->quote` + `chainB: quote->base`
- 自动选收益最高的闭环

## 为什么这样设计（维护视角）

### 1) 配置优先，算法复用

新增固定路径时，优先通过 `FIXED_PATH_RULES` 扩展，而不是改搜索代码。

优点：

- 维护成本低
- 行为一致
- 更容易做回归测试

### 2) 规则边参与计算、展示层隐藏

等价值币映射对搜索有帮助，但展示上噪声大。将“计算”和“展示”拆开处理，兼顾：

- 搜索可达性
- 页面可读性

### 3) 去重在算法层做，而不是渲染层做

闭环去重放在 `findTopCycles()` 内部，能避免：

- 排序前重复污染候选集
- 前端各处重复写去重逻辑

## 当前已知限制（升级时重点关注）

### 1) 收益率是理论值

当前 `profitRate` 基于报价连乘，不包含：

- Gas
- 滑点
- 跨链桥费用/时间成本
- 交易额度限制（深度）

### 2) 路径深度上限为 4（规则边也占步数）

当前规则边会占用 `maxDepth`。如果后续希望“规则边不占步数”，需要修改 DFS 深度计数逻辑，而不是仅改展示。

### 3) 固定路径仅支持 `steps: 2`

目前 `findBestFixedPath()` 只处理 2-step 闭环。若未来要支持多腿固定策略（例如稳定币经中间资产），需要新增规则模式。

## 后续升级建议（按优先级）

### P1（易做、收益高）

- 给固定路径规则增加 `allowedChains` 白名单（限制自动跨链枚举范围）
- 给固定路径规则增加 `minProfitRate`（避免展示过小机会）
- 面板中展示“路径来源/置信度”字段（例如是否含规则边）

### P2（功能增强）

- 规则边不计入深度（但仍参与闭环）
- 加入基础成本模型（gas / bridge fee 的估算）
- 支持固定路径多腿模板（`steps > 2`）

### P3（精度升级）

- 按金额分档重新报价（考虑滑点）
- 引入链上流动性深度/实际成交可行性过滤
- 按成交金额输出“预估净收益”而非纯比率

## 新增固定路径的维护规范（建议）

新增一条固定路径时，优先遵循：

1. 尽量先用配置表达，不改算法
2. 若是“固定币种、链不固定”场景，优先用 `crossChain: true`
3. 若需要限定链范围，再扩展规则字段（如 `allowedChains`）
4. 同步补充 `tests/arb-paths.test.js` 覆盖新规则行为
